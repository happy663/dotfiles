# PR説明文作成コマンド

指定されたissueを元に、簡潔で構造化されたPR説明文を作成してください。

## スタイル要件

- **簡潔性**: 冗長な説明は避け、要点のみを端的に表現
- **文体**: エンジニア向けの実務的な表現を使用
- **明確性**: 一文一義で、事実とデータに基づいて記述
- **敬称**: 不要な敬称や過度な丁寧語は使わない

## PR構成

以下のセクション構造で作成してください:

### ref
- 関連issueのリンク

### 背景
- 問題の事実のみを記述
- 調査結果を箇条書きで簡潔に
- 調査の網羅性が伝わるように（例: フラウド調査も実施済み、など）

### 目的
- この変更で達成したいこと（What）
- 「〜を実現する」「〜を改善する」など端的に
- 具体的な実装内容は「やったこと」セクションへ

### やったこと
- 実装した内容を箇条書きで端的に
- 技術的な変更内容を明確に

### 根拠（データがある場合）
- 数値や閾値の選定理由
- データ分析結果
- 算出ロジックを簡潔に説明
- 詳細資料へのリンク

### レビューしてほしいポイント
- レビュアーに判断を仰ぐ具体的な観点
- 疑問形で端的に

## 避けるべき表現

- ❌ 「〜することができます」→ ⭕ 「〜する」
- ❌ 「〜を考慮し、」→ ⭕ 「〜のため、」または削除
- ❌ 「〜によって〜することで〜を実現する」→ ⭕ 「〜を実現する」
- ❌ 「具体的には:」の後に冗長な説明 → ⭕ 必要最小限の情報のみ

## 例

良い例:
```
## 目的

不正クリック判定処理が正常に動作している状況下で、
誤検知によるアラート頻発を抑制しノイジーな状況を改善する
```

悪い例:
```
## 目的

不正クリック判定処理が正常に動作している状況下で、誤検知によるアラート頻発を抑制する。

具体的には:
- 現在の不正クリック発生状況(平均33,000件/時間)を考慮し、閾値を30,000から90,000に引き上げ
- 本来監視したい「処理遅延の予兆」を適切に検知できるようにする
- アラート疲れを防ぎ、真に対応が必要な状況を見逃さないようにする
```

## 実行方法

このコマンド実行後、issue番号または関連情報を提供してください。
